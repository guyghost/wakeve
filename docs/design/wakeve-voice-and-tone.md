# Wakeve Voice and Tone

## Voice
Wakeve sounds human, calm, warm, and clear. On iOS, it should feel like a capable friend helping a group get organized, not a corporate workflow tool and not a playful toy.

## Rules
- Prefer clear action labels over cleverness.
- Use warmth in content moments, not in every system control.
- Keep errors direct and actionable.
- Use short sentences on compact surfaces.
- Avoid pressure, hype, and childish language.
- Name the user benefit, not the underlying technology.
- Keep French and English labels equivalent in intent, not literally word-for-word.
- Reserve expressive wording for onboarding, empty states, success messages, and editorial cards.
- Keep navigation, permissions, deletion, privacy, payments, and confirmations neutral.

## UI Layer Copy
The UI layer should stay conventional:
- `Ajouter`
- `Modifier`
- `Supprimer`
- `Partager`
- `Rechercher`
- `Fermer`
- `Retour`

These labels should not be replaced by branded phrases.

Critical actions should include the object when ambiguity is possible:
- `Confirmer la date`
- `Lancer le sondage`
- `Supprimer l'événement`
- `Partager l'invitation`
- `Tout marquer comme lu`

## Content Layer Copy
The content layer can carry Wakeve personality:
- `Prochain moment`
- `Inviter le groupe`
- `Préparer la soirée`
- `Le moment prend forme.`
- `Tout le monde trouve sa place.`
- `La table se prépare.`
- `Le groupe se met en route.`

Use these only when context makes the meaning obvious.

Avoid using `Créer un moment` as the primary creation CTA. Use `Créer un événement` for the main action, then allow `moment` in supportive copy where the event context is already obvious.

## Product Vocabulary
Use `événement` / `event` for the primary object in critical flows. Use `moment` only for emotional or editorial references.

Use `option` / `option` for visible destination, lodging, budget, or date alternatives. Keep `scénario` / `scenario` for technical documentation, internal architecture, and analytics unless the user must understand a formal comparison model.

Use `sondage` / `poll` for the date-voting process. Use `vote` / `vote` for the action or submitted response.

Use `participant` once a person is part of the event. Use `invité` / `guest` or `invitee` before the person has responded.

For AI-assisted surfaces, prefer:
- `Préparer des suggestions`
- `Proposer des options`
- `Préparer un brouillon`
- `Résumer l'événement`
- `Créer une liste`

Avoid visible labels such as:
- `Assistant IA`
- `Générer`
- `Automatiser`
- `Smart`

## Empty States
Good empty states:
- explain what is missing
- keep the tone calm
- offer one clear next action
- avoid feature tours

Example:
`Aucun événement à venir. Créez un événement quand vous êtes prêt.`

Primary action:
`Créer un événement`

## Errors
Errors should be plain:
- what happened
- what the user can do next
- whether work is saved or queued

Example:
`Impossible de synchroniser pour l'instant. Tes changements restent enregistrés sur cet appareil.`
